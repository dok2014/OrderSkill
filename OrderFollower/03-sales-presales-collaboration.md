# 销售×售前协同工作策略

## 🎯 核心理念：销售控场，售前输出，互相补位，绝不拆台

销售和售前不是甲乙方关系，是同一个战壕里的两个射手：
- **销售**负责判断"打谁、什么时候打、打完怎么收"
- **售前**负责判断"怎么打最准、用什么弹、能不能打中"

两人合力，一击必中。

## 📋 核心协同规则（快速参考）

**销售 × 售前协同规则**（当用户询问售前阶段相关问题时启用）：
- **协同前**：确认客户信息完整交底、目标对齐、分工定界、口径统一、会前彩排
- **会前24h**：完成尽调 → 同步售前 → 对齐目标分工 → 定稿材料
- **会中**：销售控场挖需求 → 售前技术输出 → 销售收口锁下一步
- **会后24h**：输出纪要+方案调整+下一步计划
- **绝对禁忌**：销售不乱答技术/不乱承诺交付周期；售前不越界谈价格/商务；双方不在客户面前拆台/暴露内部分歧

## 一、角色分工与边界

### 1.1 销售的职责

| 职责 | 具体内容 | 检查标准 |
|------|----------|----------|
| 客户经营权 | 判断机会价值、决策链、推进节奏 | 能画出完整的决策链地图 |
| 需求挖掘 | 用SPIN提问法挖掘业务痛点 | 客户说出具体业务痛点 |
| 关系经营 | 建立高层关系、培养Champion | Champion主动提供内部信息 |
| 商务推进 | 报价、谈判、合同、回款 | 签约且利润率达标 |
| 进度控场 | 设锚点、跟进度、管预期 | 客户不追进度，等你的汇报 |

### 1.2 售前的职责

| 职责 | 具体内容 | 检查标准 |
|------|----------|----------|
| 技术验证 | 评估技术可行性、风险、方案路径 | 给出明确的技术可行性判断 |
| 方案设计 | 设计解决方案、技术架构 | 方案通过客户技术评审 |
| 技术交流 | 进行技术演示、解答技术问题 | 客户技术决策者认可 |
| 竞品分析 | 技术维度对比竞争方案 | 能说清3个以上差异化优势 |
| 交付评估 | 评估交付周期、资源、风险 | 交付周期和成本的估算偏差<20% |

### 1.3 边界红线

**销售绝对不做的：**
- 不乱答技术问题（"我们肯定能做"）
- 不乱承诺交付周期（"两个月肯定上线"）
- 不替售前做技术判断（"我回去确认一下"才是正确回答）

**售前绝对不做的：**
- 不越界谈价格和商务条件
- 不替销售做客户关系判断
- 不在客户面前暴露内部分歧
- 不替销售做承诺（"这个我们后面可以再谈"→ 越界）

## 二、协同节奏：按项目阶段对齐

### 2.1 协同前：信息交底（客户首次技术交流前）

| 事项 | 销售提供 | 售前确认 |
|------|----------|----------|
| 客户背景 | 公司规模、行业地位、业务痛点 | 行业技术趋势、常见痛点 |
| 决策链 | EB/TB/UB是谁，各人关注点 | 技术决策者的技术偏好和背景 |
| 会议目标 | 这次会议要达成什么（验证需求？展示能力？锁定下一步？） | 技术层面需要展示什么 |
| 分工定界 | 谁开场、谁主讲、谁收尾 | 技术部分讲多深、讲多久 |
| 口径统一 | 预算区间、时间表、承诺底线 | 技术可行性结论、不能承诺的边界 |

### 2.2 会前24小时：最终对齐

```
检查清单：
□ 尽调完成，客户信息同步给售前
□ 会议目标对齐（主目标 + 备选目标）
□ 分工确认：销售负责哪些环节、售前负责哪些环节
□ 材料定稿：演示内容、方案要点、Q&A预案
□ 红线确认：哪些话不能说、哪些承诺不能做
□ 应急预案：客户问出售前不在场的技术问题，销售怎么接
```

### 2.3 会中：角色分工表

| 环节 | 主导 | 动作 | 另一方配合 |
|------|------|------|------------|
| 开场破冰 | 销售 | 介绍双方、定调会议目的 | 售前适时补充客户关切 |
| 需求挖掘 | 销售 | 用SPIN提问引导客户说出痛点 | 售前观察记录技术需求细节 |
| 技术展示 | 售前 | 方案演示、技术答疑 | 销售控制时间、控制节奏 |
| 商务讨论 | 销售 | 预算、时间表、下一步 | 售前不主动发言，除非被问到 |
| 收尾锁约 | 销售 | 总结要点、确认下一步行动 | 售前确认技术可行性承诺 |

**关键原则：**
- 销售控场——什么时候切换话题、什么时候深入、什么时候收住，销售决定
- 售前输出——技术问题售前权威回答，销售不抢答不补充
- 互相补位——销售卡壳时售前帮接，售前偏技术时销售拉回业务

### 2.4 会后24小时：闭环

| 事项 | 负责人 | 产出 | 检查标准 |
|------|--------|------|----------|
| 会议纪要 | 销售 | 关键决策、待确认事项、下一步计划 | 24小时内发出 |
| 方案调整 | 售前 | 根据客户反馈调整技术方案 | 48小时内完成 |
| 下一步计划 | 销售 | 明确时间、形式、参与人 | 客户确认接受 |
| 内部同步 | 销售+售前 | 同步进展、更新风险评估 | 不超过48小时 |

## 三、售前资源调度策略

### 3.1 核心矛盾

售前资源是稀缺的，但客户需求是无限的。正确的做法不是"尽量满足"，而是**精准投入**。

### 3.2 资源调度原则

| 原则 | 具体做法 | 检查标准 |
|------|----------|----------|
| 先评估再投入 | 动用售前资源前，先完成机会评估（对照 skill-opportunity-evaluation.md） | 机会评估Go才投入 |
| 分段投入 | 不一次性投入全部售前资源，按阶段逐步加码 | 每阶段确认后再进入下一阶段 |
| 显性化投入 | 向客户明示资源投入的分量，但不诉苦 | 客户感知到你在为他的项目投入资源 |
| 退出止损 | 对照 skill-pre-sales-exit.md，触发退出条件及时释放资源 | 不沉没成本拖住售前 |

### 3.3 资源申请流程

```
销售发起资源申请 → 填写以下信息：
□ 客户名称、项目背景
□ 当前阶段及推进目标
□ 需要售前做什么（评审/方案/交流/驻场）
□ 预计投入时长
□ 机会评估结论（Go/观察/风险点）
→ 售前负责人评估 → 分配资源或要求补充信息
```

### 3.4 资源冲突处理

当多个项目同时需要售前资源时，按以下优先级排序：

| 优先级 | 判断标准 | 示例 |
|--------|----------|------|
| P0 紧急 | 签约前最后冲刺，且赢面>60% | 客户本周决策，需技术终审 |
| P1 重要 | 进入方案证明期，且有Champion支持 | 客户约了技术交流，Champion已确认参加 |
| P2 常规 | 机会验证阶段，需初步技术评估 | 首次客户会议前技术准备 |
| P3 观察 | 信息不充分，先做最小投入 | 客户态度不明确，先远程支持 |

## 四、常见协同问题与解决方案

### 4.1 销售催售前出方案，售前来不及

**场景：** 客户催方案，销售承诺了交付时间，但售前资源排不开。

**正确做法：**
1. 销售**不要当场承诺**方案交付时间，改说"我回去跟技术团队确认一下时间"
2. 销售回到内部后，按资源申请流程正式提需求
3. 如果确实排不开，销售对客户说：**"为了保证方案质量，我们需要X天，这个时间是为了确保给出的方案是经过验证的"**
4. 同时给客户一版框架（参考之前"分段交付"原则），让他先看方向

**绝对不要：** 回来催售前"客户很急你赶紧搞一下"——这只会导致匆忙产出的低质量方案。

### 4.2 售前方案太技术化，客户听不懂

**场景：** 售前出了一份技术含量很高的方案，但客户决策者是业务出身。

**正确做法：**
1. 销售在会前对齐时明确告知客户决策者背景
2. 要求售前准备两个版本：技术详细版 + 业务摘要版
3. 会上销售负责把技术语言翻译成业务价值

**关键话术：**
> "我帮您翻译一下——刚才说的X架构，对您的业务意味着Y，具体来说就是……"

### 4.3 客户现场问到售前答不出的问题

**场景：** 客户问了某个超出售前准备范围的问题。

**正确做法：**
1. 售前诚实回答："这个问题我需要回去验证一下，X号给您明确回复"——比乱答强一百倍
2. 销售接场："这个问题很关键，我们回去确认后第一时间给您回复，不耽误您的判断"
3. 会后24小时内必须给回复

**绝对不要：** 两个人互相看、都不说话、现场沉默。

### 4.4 售前和销售对项目判断不一致

**场景：** 销售认为值得投入，售前认为技术风险太大。

**正确做法：**
1. 在内部讨论，不在客户面前暴露分歧
2. 售前给出具体的技术风险点，销售评估业务影响
3. 共同决策：Go（风险可控）/ 观察（追加信息后再判断）/ No-Go（风险不可接受）
4. 如果无法达成一致，升级到上级决策

## 五、业绩归属与互评机制

### 5.1 业绩归属

| 场景 | 归属原则 |
|------|----------|
| 签约项目 | 销售为主，售前协助按贡献比例分配 |
| 技术原因赢单 | 售前贡献占比提高 |
| 技术原因丢单 | 售前复盘，但不单独问责 |
| 商务原因丢单 | 销售复盘，但不单独问责 |

### 5.2 互评维度

**销售评售前：**
- 响应速度
- 方案质量（是否通过客户评审）
- 现场配合度
- 风险预警及时性

**售前评销售：**
- 信息交底完整性
- 需求挖掘深度
- 现场控场能力
- 资源申请合理性

## 六、专业承诺与委婉拒绝框架

### 6.1 核心逻辑置换

当面对客户或内部协作方的请求，明知难以满足但不忍直接拒绝时，必须使用专业承诺框架：

| 旧逻辑（不专业） | 新逻辑（专业可靠） |
| :--- | :--- |
| 不拒绝 = 有机会成功 = 不让对方失望 | 清晰界定责任边界 + 管理预期 = 控制结果 = 真正的可靠 |
| 帮助对方 = 替对方承担所有风险 | 帮助对方 = 诚实地评估可行性，让对方有时间找备选 |
| 拒绝 = 伤害关系 | 拒绝 = 条件谈判，把"不"转化为"可以，但需要……" |

**专业提醒：** 一次"勉强答应后的失败"带来的失望，远大于"提前告知风险时的小失望"。

### 6.2 标准话术公式（三步走）

当预感"大概率做不到"时，不要直接说"行"或"不行"，使用以下结构：

> **承接情绪 + 量化门槛 + 转换责任主体**

**完整模板：**
> "我完全理解你对【对方的核心需求】的迫切需求。  
> 这件事如果要做到，需要满足【条件A、B、C】，其中【B和C】目前我无法单独控制。  
> 我可以先投入时间尝试攻克【A部分】，但为了不耽误你的最终目标，我们需要设定一个检查点：在【具体日期】之前，如果【关键条件】没有达成，我会提前告知你，届时你需要启动备选方案。你看这样可以吗？"

### 6.3 销售场景化话术库

| 场景 | 话术 | 适用对象 |
| :--- | :--- | :--- |
| **客户催方案，但资源紧张** | "这个方案如果要做到您期望的效果，需要【X天】的售前深度投入。目前我只能先提供框架和关键假设，如果您认可方向，我们再安排详细设计。这样既不影响您判断，也不浪费双方资源。" | 客户 |
| **售前资源不足，但客户要求紧急** | "我理解您希望尽快看到完整方案。不过要保证方案质量，需要【售前专家】投入【3天】时间。我可以先协调【1天】做初步评估，如果可行我们再安排详细设计。同时建议您也同步了解其他方案选项，确保不耽误进度。" | 客户/内部 |
| **技术可行性不确定** | "根据我的初步判断，这个需求在技术上实现需要满足【A、B、C】三个条件。目前【B条件】存在不确定性。我建议我们先做一个小范围POC验证，如果验证通过再推进完整方案。这样风险可控，也不耽误您后续计划。" | 客户/售前 |
| **客户要求超范围承诺** | "我理解您希望有明确保障。不过按照行业标准，这类项目的成功需要双方共同配合。我们可以承诺的是【我们的责任范围】，同时需要您这边确保【客户配合事项】。这样责任清晰，结果可预期。" | 客户 |
| **内部资源协调困难** | "这个需求我评估需要【X资源】支持。目前资源排期紧张，我可以先争取【部分资源】做前期工作，同时我们设定一个检查点：如果【关键资源】在【时间点】前无法到位，我会提前告知，届时我们需要调整计划或寻找替代方案。" | 内部团队 |

### 6.4 底线原则

1. **永远不用"我试试"逃避明确表态**："试试"必须附上时间、标准和失败后的备选方案
2. **接受时必须说明失败通知机制**：明确告知"如果失败，我会在什么时间点、用什么方式通知你"
3. **主动帮对方想备选方案**："如果我不行，你可以找XXX，他擅长这个"比硬撑更能赢得尊重
4. **量化风险而非模糊描述**：不说"可能有问题"，而说"根据历史数据，这类情况有30%的概率会出现XX问题"

### 6.5 与售前协同的结合应用

#### 对客户使用：
- **方案交付**：分段交付而非一次性承诺
- **时间承诺**：给阶段锚点而非最终截止日
- **资源投入**：显性化投入成本，建立平等对话

#### 对内部使用：
- **资源申请**：明确成功条件和备选方案
- **风险预警**：提前告知而非事后解释
- **责任界定**：清晰划分销售与售前责任边界

### 6.6 效果检查标准

每次使用承诺框架后，检查：
- ✅ 是否清晰给出了"什么条件下我会停止努力"？
- ✅ 对方是否理解了成功概率不高，并有心理准备？
- ✅ 是否设定了明确的检查点和通知机制？
- ✅ 比起一口答应然后搞砸，这次是否让对方更信任？

如果四个都是"是"，则框架使用成功。记住：**专业人士不负责满足所有人的情绪，只负责提供可执行的真相。**

### 6.2 飞龙虾在售前协同中的核心作用

#### 策略建议生成
```yaml
输入: 客户背景 + 会议目标 + 售前资源情况
输出:
  - 最优分工方案（谁主讲、谁控场、谁收尾）
  - 风险预警清单（技术不可承诺点、商务敏感点）
  - 应急预案（客户问出超范围问题的应对方案）
  - 成功标准（本次会议要达成的具体目标）
```

#### 实时协同支持
